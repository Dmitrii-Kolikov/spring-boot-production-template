## 📥 REST-слой: Логирование, Маскирование и Валидация

Для автоматического контроля, логирования и защиты входящего REST-трафика используется специализированный компонент **`RestLoggingFilter`** (компонент `OncePerRequestFilter` с наивысшим приоритетом выполнения).

### ⚙️ Спецификация работы логгера
* **Поддержка Kotlin Coroutines (Async Dispatch)**: Корректно обрабатывает асинхронный запуск потоков без дублирования записей. Входящий запрос логируется сразу (`Rest request call`), а ответ (`Rest response call`) — строго после того, как корутина вернет данные в поток ответа.
* **Маскирование данных**: Динамически определяет эндпоинты через `RequestMappingHandlerMapping` и скрывает чувствительные поля на основе аннотации `@MaskedFields` над методами контроллеров.
* **Раздельные блоки**: Запрос и ответ выводятся изолированными структурами, каждая из которых содержит заголовки, тело и сквозной `REST_ID` для точного отслеживания времени обработки вызова сервером.

### 🛡️ Валидация по JSON-схемам (`@Valid`)
В контрактах контроллеров настроена строгая валидация структуры данных на соответствие заданным JSON-схемам:
* **Валидация запросов и ответов**: Аннотация `@Valid` применяется **как к входящим RequestDTO, так и к исходящим ResponseDTO** (возвращаемым значениям методов контроллера).
* **Контроль структуры**: Это гарантирует, что клиент не сможет прислать некорректный JSON, а сервер — отдать наружу ответ, не соответствующий утвержденной схеме контракта.

### 🚨 Технические зависимости
* **Обработка ошибок**: Для перехвата исключений и корректного вывода тела ответа при сбоях фильтр требует обязательного наличия настроенного `@RestControllerAdvice`.

## 🏗️ Особенности работы с БД: Управление транзакциями и явный Flush

В нашем микросервисе используется гибридный подход к работе с Hibernate. Мы **отказались от неявного закрытия транзакций** в пользу ручного управления сбросом буфера (*Explicit Flush*).

> [!IMPORTANT]  
> В проекте строго запрещено изменять сущности в памяти и надеяться, что Hibernate сам запушит изменения в БД в конце метода. Все изменения должны отправляться в базу явно.

### 🛑 Почему мы пишем именно так?

1. **Защита пула соединений (Connection Pool)**  
   Все долгие интеграционные вызовы (внешние REST/gRPC API) выполняются **вне** аннотации `@Transactional`. Мы открываем транзакции точечно, только на момент непосредственной записи данных в БД. Это защищает приложение от исчерпания соединений и зависания базы под высокой нагрузкой.

2. **Честные метрики и логи выполнения**  
   Наш логгер `DbCompactLogging` перехватывает вызовы репозиториев и замеряет точное время выполнения SQL-запросов. Если полагаться на автоматический коммит Спринга, Hibernate отправит все `UPDATE` запросы уже после завершения метода сервиса. Из-за этого логгер фиксировал бы ложные `0 ms`, а реальная тяжелая нагрузка на БД оставалась бы «слепой».

### 🛠️ Инструкция для разработчиков

При написании кода в слое БД следуйте четырем железным правилам:

* **Для создания новых записей (`INSERT`)**  
  Используйте `repository.saveAndFlush(newEntity)`.

* **Для пакетных операций (`Batch INSERT/UPDATE`)**  
  Сначала наполняйте списки или вызывайте `repository.saveAll(listOfEntities)`, и строго после этого делайте один общий `repository.flush()` для фиксации батча.

* **Запрет на использование `.copy()` при обновлениях**  
  При изменении сущностей **нельзя** использовать метод `.copy()` у Kotlin Data-классов. Метод `.copy()` создаёт абсолютно новый объект в памяти. Из-за этого объект вылетает из контекста Hibernate (становится Detached), ломается оптимизация `@DynamicUpdate`, и Hibernate выполняет лишний скрытый `SELECT`, перезаписывая абсолютно все колонки таблицы целиком.

### 📝 Примеры кода

#### 1. Правильное обновление (Single UPDATE)
```kotlin
val position = positionsRepository.findByIdOrNull(id)
if (position != null) {
    // 1. Меняем поля существующего объекта (мутация на месте)
    position.toUpdatePositionFromIndividualUpdate(rq, regionCode, rs) 
    
    // 2. Явный пуш в базу + честный замер времени аспект-логгером
    positionsRepository.flush()         
}
```

#### 2. Правильная работа со списками (Batch)
```kotlin
val employeesToSave = mutableListOf<EmployeeEntity>()

rs.employee.forEach { dto ->
    val existingEmployee = existingEmployeesMap[dto.employeeNumber]
    if (existingEmployee != null) {
        existingEmployee.toUpdateEmployeeFromIndividualUpdate(individualId, dto) // Мутируем
    } else {
        employeesToSave.add(rs.toSaveEmployeeFromIndividualUpdate(dto)) // Создаем новый
    }
}

// Если были новые записи - передаем их в saveAll
if (employeesToSave.isNotEmpty()) {
    employeeRepository.saveAll(employeesToSave) 
}

// Один общий flush на весь блок для экономии ресурсов и красивого лога
employeeRepository.flush() 
```

## 📡 Интеграционные логи: Кастомное логирование Feign-клиентов

Для внешних вызовов используется общая конфигурация (`CustomFeignClientLogger`), которая объединяет запрос и ответ в **один компактный блок** (`Integration call`) и оптимизирует размер логов.

### ⚙️ Спецификация работы логгера
1. **Маскирование данных**: Поточно скрывает приватные и чувствительные данные на основе аннотации `@MaskedFields` над методами Feign-клиентов (через потоковый JSON-маскер).
2. **Фильтрация и сокращение логов (`isRestShortLogs`)**: При включенном флаге удаляет из тела ответа лишние поля, не входящие в официальную структуру DTO метода.
3. **Объединенный формат**: Предотвращает дублирование и разрыв записей в консоли. Запрос и ответ выводятся одной атомарной структурой, привязанной к сквозному `REST_ID`.

### 🚨 Важные требования при разработке
* **Конфигурация Jackson**: Для корректной фильтрации структуры DTO в `application.yaml` обязательно должно быть отключено падение при неизвестных свойствах:
```yaml
  spring:
    jackson:
      deserialization:
        fail-on-unknown-properties: false
  ```
* **Запрет лишних полей**: Чтобы логи Feign-клиентов гарантированно были сокращенными и не содержали избыточных данных от внешних систем, в JSON-схемах **обязательно** нужно указывать параметр:
  ```json
  "additionalProperties": false
  ```
